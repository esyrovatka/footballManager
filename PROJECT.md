# Football Manager Simulator

Веб-приложение — симулятор менеджера футбольной команды. Несколько живых игроков подключаются к одной лиге, остальные места занимают боты. Каждый менеджер отвечает за покупку/продажу игроков, стартовый состав, замены по ходу матча, тактику и финансы клуба.

---

## 1. Технологический стек

| Слой | Технология |
|---|---|
| Фронтенд + бэкенд | Next.js 15 (App Router) + TypeScript |
| База данных | PostgreSQL |
| ORM | Drizzle ORM |
| Авторизация | Auth.js (NextAuth v5) — Google + GitHub OAuth |
| Real-time | Server-Sent Events (SSE) |
| Воркер симуляции | Отдельный Node.js процесс (тикер) |
| Планировщик туров | Vercel Cron / pg_cron |
| UI | Tailwind CSS + shadcn/ui |
| Деплой | Vercel (web) + Railway/Fly.io (воркер + БД) |

**Адаптивность**: desktop и mobile равноправно.

---

## 2. Структура лиги

- **1 лига = 8–10 команд**, круговой турнир в 2 круга = 18 туров за сезон
- **Лиги изолированы** друг от друга: своя БД-копия игроков, свой сезон, своя экономика
- **Общая админская база игроков** — при создании лиги делается snapshot, дальше игроки живут независимо в каждой лиге
- **1 тур = 1 день** реального времени. Сезон длится ~18 дней
- Каждый тур стартует в фиксированное время (по умолчанию 20:00 по таймзоне лиги)
- **Создавать лиги может только админ**
- Менеджеры присоединяются по приглашению (email или код лиги)
- Пустые слоты автоматически заполняются ботами

---

## 3. Авторизация

- Только OAuth: Google и GitHub
- Без паролей, без email-верификации
- Роли: `user` (менеджер) и `admin`

## 4. Игроки (футболисты)

### Атрибуты
Простая модель — 3–5 общих характеристик:

| Атрибут | Описание |
|---|---|
| `attack` | Атакующие действия (удар, дриблинг) |
| `defense` | Оборонительные действия (отбор, блок) |
| `speed` | Скорость и физика |
| `goalkeeping` | Только для вратарей |
| `overall` | Итоговый показатель (производный или хранимый) |
| `position` | GK / DEF / MID / FWD |
| `age` | Возраст |

### Прогрессия по возрасту
Каждый игрок при создании получает скрытые параметры:
- `peak_age`: 24–28 (случайно)
- `growth_rate`: от −1.5 до +2.5 (нормальное распределение)
- `decline_rate`: от −0.5 до −2

Раз в сезон (в межсезонье) пересчёт:
- `age < peak_age`: `overall += growth_rate ± random(0.5)`
- `peak_age ≤ age ≤ peak_age + 2`: `overall += ±random(0.3)` (плато)
- `age > peak_age + 2`: `overall += decline_rate ± random(0.5)`

Итог: разные игроки имеют разные карьерные траектории.

### Управление базой
- Игроков в админскую базу добавляет **только админ** (через админ-панель)
- В базе хранятся «шаблоны» (`player_templates`)
- При создании лиги создаётся snapshot — копия в `league_players` со своим состоянием

## 5. Клуб

- **Ростер**: 11 в основе + 7 в запасе = 18 в матчевой заявке
- **Бюджет** (€ миллионы)
- **Менеджер** — пользователь или бот
- **Тактика**: формация (4-4-2, 4-3-3, 3-5-2 и т.д.) + общий стиль (атакующий / балансированный / защитный)

---

## 6. Матч-день (ключевая механика)

### Расписание
- Тур запускается автоматически по расписанию (cron job)
- Все матчи тура стартуют одновременно
- До дедлайна менеджер обязан выставить стартовый состав и правила замен (иначе используется состав с прошлого тура)

### Длительность матча
- **10–15 минут реального времени** на матч (90 in-game минут)
- Соотношение: ~10 секунд реального времени = 1 in-game минута

### Симуляция
- **Текстовая лента событий**: голы, опасные моменты, фолы, карточки, замены, ключевые пасы
- Match Engine — детерминированный генератор, на вход: составы обеих команд + тактика + seed
- Match Runtime — воркер с тикером каждые ~10 секунд продвигает все активные матчи на 1 минуту
- Real-time доставка событий клиенту через **Server-Sent Events**

### Live-режим (присутствие менеджера на матче)
- Если менеджер заходит во время матча — может делать замены и менять тактику «на лету»
- Команды записываются в таблицу `live_directives`, runtime применяет их на следующем тике
- Если не зашёл — работают **заранее заданные правила замен** (например: «если 0:0 на 70', выпустить запасного нападающего», «если устал X — заменить на Y»)
- Если нет ни менеджера, ни правил — система делает разумные авто-замены (уставших, травмированных)

### Дисциплина
- Жёлтые карточки, красные карточки
- Накопленные дисквалификации (например, 5 жёлтых = пропуск тура)

---

## 7. Трансферы

- **Между клубами через офферы**: можно сделать предложение любому клубу за его игрока, клуб (или бот) принимает/отказывает
- **Пул свободных агентов**: игроки без контракта, доступные за фиксированную цену
- **Цена игрока**: формула на основе overall, возраста и потенциала (примерно `(overall^2 / 80) * age_modifier`)
- **Принятие оффера ботом**: зависит от соотношения цена/рыночная стоимость + наличия замены в составе
- **Открыто на решение**: трансферные окна или круглогодично (решим в фазе 3)

---

## 8. Финансы

Простая модель:

### Расходы
- Зарплаты игроков (еженедельные, ~0.5% от трансферной стоимости)
- Трансферные платежи

### Доходы
- За каждый тур: победа €1M, ничья €300K, поражение €0
- В конце сезона по итоговому месту:
  - 1-е: €15M
  - 2-е: €10M
  - 3-е: €7M
  - Далее по убывающей до €1M за последнее

### Стартовый бюджет
- €50M на клуб при создании лиги

### Логика при отрицательном балансе
- Можно уйти в небольшой минус
- При `budget < -€10M`:
  - Блокируются исходящие трансферы
  - Шанс принятия оффера на твоих игроков ботом +30% (вынужденные продажи)

---

## 9. Боты

- **MVP-минимум**: всегда выставляют состав по `overall`, выбирают базовую формацию (4-4-2), иногда покупают игрока если в составе явная дыра, отвечают на офферы по простой формуле
- Архитектура должна позволять усложнить логику позже (разные «характеры», стратегии трансферов)

## 10. Лента новостей лиги

Публичные события лиги:
- Совершённые трансферы
- Результаты тура (с ключевыми моментами)
- Сенсации (победа аутсайдера над лидером)

## 11. Архитектура

### Компоненты
1. **Next.js web app** (Vercel)
   - SSR / API routes / Server Actions
   - Auth, страницы менеджера, админка
   - SSE-эндпоинт для трансляции матчей
2. **PostgreSQL** (Neon / Supabase / Railway)
3. **Match Runtime** — отдельный Node.js воркер
   - Каждые ~10 сек: для всех матчей в статусе `running` продвинуть `current_minute += 1`
   - На каждом тике: применить очередное событие из заранее сгенерированной ленты + любые pending `live_directives`
   - По окончании матча: статус → `finished`, обновить таблицу
4. **Cron Job** (Vercel Cron / pg_cron)
   - Раз в сутки в установленное время лиги: создать матчи нового тура, статус `running`
   - В конце сезона: пересчёт прогрессии, генерация нового сезона

### Поток матча (life-cycle)
```
Daily 20:00 → cron создаёт записи матчей (status: running, scheduled_at)
            → Match Engine генерирует event log с seed (детерминированно)
            → Match Runtime начинает тикать
            → Клиенты подключаются к SSE и получают события минута-в-минуту
            → Менеджер может слать live_directives (substitutions, tactic change)
            → Через ~12 минут матч в статусе finished
            → News feed обновляется, финансы пересчитываются
```

## 12. Схема базы данных (черновик)

```
users
  id, email, oauth_provider, oauth_id, display_name, is_admin, created_at

player_templates              -- админская база
  id, name, position, base_overall, age, peak_age, growth_rate, decline_rate,
  nationality, attributes (jsonb)

leagues
  id, name, status (setup|active|finished), season_number, current_round,
  match_time_local, timezone, created_at

clubs
  id, league_id, name, budget, manager_user_id (nullable), is_bot, founded_round

league_players                -- инстансы игроков в конкретной лиге
  id, league_id, club_id, template_id, current_overall, current_age,
  yellow_cards, suspended_until_round, contract_salary, contract_until_season

seasons
  id, league_id, season_number, status (active|finished), started_at, finished_at

matches
  id, season_id, round, home_club_id, away_club_id, scheduled_at, status,
  home_score, away_score, current_minute, engine_seed, finished_at

match_events
  id, match_id, minute, type (goal|foul|yellow|red|sub|chance|...),
  club_id, player_id (nullable), description, payload (jsonb)

lineups
  id, match_id, club_id, formation, style (attack|balanced|defense),
  starters (player_id[]), subs (player_id[]), submitted_at

sub_rules
  id, lineup_id, condition (jsonb), player_in_id, player_out_id, priority

live_directives
  id, match_id, user_id, type (sub|tactic_change), payload (jsonb),
  created_at, applied_at (nullable)

transfer_offers
  id, league_id, from_club_id, to_club_id, player_id, amount,
  status (pending|accepted|rejected|withdrawn), created_at, resolved_at

free_agents
  id, league_id, player_template_id, asking_price

news_items
  id, league_id, type, payload (jsonb), created_at
```
