# Roadmap — пошаговый план реализации

Цель: довести до играбельного MVP с мультиплеером, в котором несколько живых менеджеров + боты могут пройти полный сезон с трансферами, матчами в реальном времени и прогрессией игроков.

Каждый шаг даёт **видимый прогресс** — можно остановиться и проверить.

---

## Фаза 2: Жизненный цикл лиги (3 шага)

### 2.1. Создание лиги
- [ ] В админ-панели: страница «Создать лигу»
  - Название, количество клубов (8–10), названия клубов, таймзона, время матчей
  - Выбор подбора игроков (вся база / случайные N / диапазон overall)
- [ ] Snapshot: при создании генерируются `league_players` для всех клубов
  - Распределение игроков по клубам: случайно с балансировкой по позициям и overall
  - Каждому клубу стартовый бюджет €50M
  - Контракты с зарплатами
- [ ] Расписание сезона: круговой турнир в 2 круга, алгоритм Round Robin
  - Записи матчей создаются заранее со статусом `scheduled`

**Результат**: можно создать лигу, увидеть клубы, ростеры и календарь матчей.

### 2.2. Приглашение менеджеров
- [ ] Админ может пригласить пользователя на конкретный клуб (по email или сгенерированной ссылке)
- [ ] Пользователь принимает приглашение → становится `manager_user_id` клуба
- [ ] Незанятые клубы помечены `is_bot = true`
- [ ] Пользователь видит список своих клубов на `/dashboard`

**Результат**: несколько живых пользователей подключились к одной лиге как менеджеры разных клубов.

### 2.3. Просмотр клуба и лиги
- [ ] Страница клуба `/leagues/[id]/clubs/[clubId]`:
  - Ростер с фильтрами по позиции
  - Бюджет, расходы на зарплаты
  - Следующий матч и расписание
- [ ] Страница лиги `/leagues/[id]`:
  - Турнирная таблица (изначально пустая)
  - Календарь
  - Список клубов с менеджерами
- [ ] Страница игрока: статы, контракт, история

**Результат**: можно ходить по клубам и видеть состояние лиги (но матчи ещё не играются).

## Фаза 3: Подготовка к матчу (3 шага)

### 3.1. Редактор состава и тактики
- [ ] Страница `/leagues/[id]/clubs/[clubId]/lineup`:
  - Drag-and-drop или клик-выбор: 11 в основу + 7 в запас
  - Выбор формации (4-4-2, 4-3-3, 3-5-2, 5-3-2, 4-2-3-1)
  - Выбор стиля: атакующий / балансированный / защитный
  - Сохранение в `lineups` (или draft до старта матча)
- [ ] Валидация: ровно 1 GK в основе, корректная формация
- [ ] Авто-предложение состава (для нового пользователя)

**Результат**: можно собрать состав на следующий матч.

### 3.2. Правила замен
- [ ] Редактор `sub_rules` на странице состава:
  - Условия: «если проигрываем после 60'», «если ведём после 75'», «если устал > X», «если получил жёлтую»
  - Действия: заменить X на Y
  - До 3 правил на матч (как реальные замены)
- [ ] Сохранение в `sub_rules`, привязанные к `lineup`

**Результат**: менеджер задаёт авто-замены даже если не зайдёт на матч.

### 3.3. Трансферный рынок
- [ ] Страница `/leagues/[id]/transfers`:
  - Список свободных агентов с фильтрами (позиция, overall, цена)
  - Поиск игроков других клубов
  - Кнопка «Сделать предложение»: вводишь сумму, отправляешь
- [ ] Обработка офферов:
  - Если бот → автоответ по простой формуле (overall_replacement_in_squad + сумма vs рыночная цена)
  - Если живой менеджер → нотификация и страница `/inbox` с принять/отклонить
- [ ] При принятии: игрок переходит, бюджеты обновляются, запись в `news_items`
- [ ] Свободные агенты — кнопка «подписать» за фиксированную цену

**Результат**: рабочий трансферный рынок, можно менять состав между турами.

## Фаза 4: Матч-день — ядро (5 шагов)

### 4.1. Match Engine (детерминированный генератор)
- [ ] Изолированная библиотека в `src/engine`
- [ ] Чистая функция:
  ```
  simulate(homeLineup, awayLineup, seed) → MatchEvent[]
  ```
- [ ] Генерация ~30–80 событий на 90 минут
- [ ] Типы событий: `chance`, `goal`, `foul`, `yellow`, `red`, `save`, `corner`, `injury`, `key_pass`
- [ ] Вероятности зависят от:
  - Соотношение overall команд по линиям (атака против защиты)
  - Стиль игры (атакующий → больше шансов и пропускных)
  - Формация (3-5-2 → больше владения в центре)
- [ ] Юнит-тесты: миллионы симуляций → разумные распределения голов (среднее ~2.5 за матч)
- [ ] Текстовые описания событий (шаблоны с именами игроков)

**Результат**: чистая функция, на вход составы — на выход правдоподобный лог матча. Можно играть «вручную» через тесты.

### 4.2. Cron-планировщик туров
- [ ] Vercel Cron / pg_cron, запускается каждый час
- [ ] Логика: для каждой активной лиги — если `now >= scheduled_at` следующего тура и он `scheduled`:
  - Зафиксировать составы (взять последний `lineup` или авто-составить)
  - Сгенерировать `match_events` через Match Engine с seed = hash(match_id)
  - Сменить статус матчей на `running`
  - Установить `started_at`
- [ ] При отсутствии lineup — авто-состав по `overall`

**Результат**: матчи стартуют автоматически в назначенное время.

### 4.3. Match Runtime (тикер)
- [ ] Отдельный Node.js процесс (`scripts/runtime.ts`)
- [ ] Каждые 10 секунд:
  - Для всех матчей `running`: `current_minute += 1`
  - Получить события этой минуты из заранее сгенерированного лога → записать в `match_events` с `revealed_at = now`
  - Применить pending `live_directives`
  - Если `current_minute >= 90`: status → `finished`, записать счёт, обновить бюджеты, начислить призовые
- [ ] Обработка одновременных матчей в одной транзакции
- [ ] Деплой воркера: Railway/Fly.io или Docker

**Результат**: матчи реально идут в реальном времени, ~12 минут на матч.

### 4.4. Live-лента матча (SSE)
- [ ] Эндпоинт `/api/matches/[id]/stream` — Server-Sent Events
- [ ] Клиент подключается → получает все события до текущей минуты + стрим новых
- [ ] Страница `/leagues/[id]/matches/[matchId]`:
  - Хедер: счёт, минута, формации обеих команд
  - Лента событий с автоскроллом
  - Статистика (удары, владение)
- [ ] При статусе `finished` — финальные данные и кнопка «протокол матча»

**Результат**: пользователи видят матч в реальном времени.

### 4.5. Live-команды менеджера
- [ ] На странице матча, если ты менеджер участвующего клуба:
  - Кнопка «Сделать замену» → выбор кто-кого → запись в `live_directives`
  - Кнопка «Сменить стиль» → запись `live_directives`
- [ ] Runtime подхватывает директивы на следующем тике, применяет к оставшимся событиям лога (например, после замены атакующий игрок чаще участвует в `chance`-событиях)
- [ ] Лимит: 3 замены за матч
- [ ] Бот-логика для матча:
  - Применяет `sub_rules` если они есть
  - Иначе делает «разумные» замены (уставший после 70', травмированный)

**Результат**: полноценный live-режим, влияние решений менеджера на ход матча.

## Критические точки риска

1. **Match Engine** — сердце игры. Если симуляция выдаёт нереалистичные результаты, играть невозможно. **Прототипировать рано** (фаза 4.1, ещё до runtime).
2. **Match Runtime** — деплой долгоживущего воркера на Vercel невозможен, нужен Railway/Fly/Render. Решить инфраструктуру до фазы 4.3.
3. **Конкурентный доступ** к live-директивам — нужны транзакции и блокировки в Postgres, чтобы две команды от менеджера не перезаписали друг друга.
4. **Авто-составы** — если менеджер не зашёл, нужен fallback. Лучше всегда хранить «последний рабочий состав» как dafault.
5. **Боты-менеджеры** — даже минимум должен быть разумным, иначе ИИ-команды будут проигрывать всегда → нет соревнования.

- Чат лиги

- Тренер и тренерский штаб
